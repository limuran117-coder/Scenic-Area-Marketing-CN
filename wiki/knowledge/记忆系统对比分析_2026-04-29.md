## 一、OpenClaw 记忆系统深度分析

### 1.1 当前架构概览

OpenClaw 采用**三层记忆架构**，结构如下：

| 层级 | 文件/位置 | 内容 | 持久性 |
|------|----------|------|--------|
| **L0** | SOUL.md / IDENTITY.md / USER.md | 角色定义、行为准则、用户信息 | 静态配置 |
| **L1** | MEMORY.md | 精选长期记忆（curated wisdom） | 持久，需手动更新 |
| **L2** | memory/YYYY-MM-DD.md | 每日工作日志 | 持久，自动追加 |
| **L3** | memory/.dreams/ | 梦境系统（短期召回+会话归档） | 半持久 |
| **L4** | wiki/ | 外部知识库（Karpathy Wiki 模式） | 持久，语义索引 |

### 1.2 各层详细分析

#### L0 — 角色与元规则层
**存储位置：** SOUL.md / IDENTITY.md / USER.md

- `SOUL.md`：人格特质、行为准则（karpathy-guidelines 整合）
- `IDENTITY.md`：身份定义（李涯/景区营销总经理）
- `USER.md`：用户信息、关键文件路径、浏览器配置

**特点：** 静态配置，Session 启动时自动加载，定义了"我是谁"的基础

**评价：** 这层设计很扎实，角色清晰度直接影响 Agent 表现。缺点是修改需要手动编辑文件，没有版本控制。

---

#### L1 — 长期记忆（Curated Memory）
**存储位置：** MEMORY.md（100 行 / 25KB 上限）

**结构：** 按类别组织（user/feedback/project/reference/insight），使用 `[type]` 标签分类

**关键设计：**
- `[feedback]`：纠错 + 确认双通道
- `[project]`：项目状态跟踪
- `[reference]`：外部系统指针
- Promoted 机制：从短期记忆提升高价值条目

**问题观察：**
- 容量限制（100 行）被反复提及但仍然频繁触发
- 提升（promotion）需要手动注释标记，不够自动化
- 内容容易过时，需要定期整理

---

#### L2 — 每日工作日志
**存储位置：** memory/YYYY-MM-DD.md

**内容：** 当日任务执行记录、系统状态、关键决策、待办事项

**读写策略：**
- 每次 Session 启动读取当天 + 昨天的日志
- 主 Session 额外读取 MEMORY.md
- 每日自动追加，无自动淘汰

**问题观察：**
- 日志量持续增长（当前已有 2026-04-19 到 2026-04-29 的文件）
- 没有自动淘汰机制，越积越多
- 搜索效率随文件数量下降

---

#### L3 — 梦境系统（Dream System）
**存储位置：** memory/.dreams/

**核心文件：**

| 文件 | 功能 | 大小 |
|------|------|------|
| `session-ingestion.json` | 跨会话轨迹索引（会话 Corpus） | 很大（数百 KB） |
| `short-term-recall.json` | 短期召回条目（带评分/标签） | ~500+ 条 |
| `phase-signals.json` | 轻量级召回索引（仅计数） | ~4000+ 条 |
| `events.jsonl` | 事件流日志 | 持续追加 |
| `daily-ingestion.json` | 每日摄取记录 | — |

**召回机制（short-term-recall）：**
- 每个条目包含：`totalScore`、`maxScore`、`recallCount`、`conceptTags[]`
- 根据 recallCount × score 计算 importance
- 低于阈值的条目会自然淘汰

**关键问题：**
- `session-ingestion.json` 持续增长但很多条目的 `lastContentLine: 0`（未被真正使用）
- 存在大量 `.deleted.` 会话文件，说明有 GC 机制但可能滞后
- `phase-signals.json` 和 `short-term-recall.json` 的关系不清晰（一个是轻量索引，一个是完整条目）
- **没有真正的遗忘机制**：梦境系统只是索引，不会删除过期的日文件

---

#### L4 — Wiki 知识库（Karpathy Wiki 模式）
**存储位置：** wiki/（含 raw/、sources/、concepts/、queries/、entities/）

**设计理念：** 将外部知识编译为可被 `memory_search(corpus=wiki)` 召回的语义索引

**目录结构：**
```
wiki/
  raw/      → 原始文档（未处理）
  sources/  → 来源标记
  concepts/ → 概念定义
  entities/ → 实体档案
  queries/  → 常见问答
  SOP/      → 标准操作程序
```

**优点：** 外部知识与内部记忆分离，知识积累可持续  
**缺点：** Ingest 流程需要手动触发，没有自动抽取机制

---

### 1.3 当前记忆系统的问题（短板分析）

| 问题 | 严重程度 | 描述 |
|------|---------|------|
| **容量管理缺失** | 🔴 高 | MEMORY.md 反复超限，没有自动压缩/promotion 机制 |
| **日志无限增长** | 🔴 高 | memory/*.md 无限追加，无淘汰/归档策略 |
| **Promoted 机制半自动** | 🟡 中 | promotion 需要手动写注释，没有自动识别高价值条目的逻辑 |
| **梦境索引膨胀** | 🟡 中 | session-ingestion.json 持续增长，很多会话数据从未被读取 |
| **召回策略不透明** | 🟡 中 | short-term-recall 的评分公式不明确，conceptTags 质量参差不齐 |
| **L0 缺乏版本控制** | 🟢 低 | SOUL.md/IDENTITY.md 修改无版本记录 |
| **跨层信息冗余** | 🟢 低 | 同一事实可能同时存在于 daily log + MEMORY.md + wiki |

## 二、业界优秀记忆系统对比

### 2.1 GenericAgent（lsdefine/GenericAgent）

**代码规模：** ~3K 行核心代码，Agent Loop 仅 ~100 行

**核心哲学：不要预设技能，靠进化获得能力**

#### L0-L4 五层记忆架构

| 层级 | 名称 | 功能 |
|------|------|------|
| **L0** | Meta Rules（元规则） | Agent 基础行为规则和系统约束 |
| **L1** | Insight Index（洞察索引） | 极简索引，用于快速路由与召回 |
| **L2** | Global Facts（全局事实） | 长期运行中积累的稳定知识 |
| **L3** | Task Skills / SOPs（任务技能） | 可复用的工作流程 |
| **L4** | Session Archive（会话归档） | 从已完成任务提炼的归档记录 |

#### 核心创新：自动 Crystallize（结晶）机制

```
[遇到新任务] → [自主摸索] → [将执行路径固化为 Skill] → [写入记忆层] → [下次同类任务直接调用]
```

**电影小镇可以借鉴的：**
- L3 技能沉淀：当前 SOP 靠手动写，但 GenericAgent 会自动从执行路径提取
- L4 会话归档：当前没有对"已完成任务"的系统性归档复盘
- **Token 高效性**：<30K 上下文 vs 当前 OpenClaw 可能超过 200K

---

### 2.2 MemGPT（大模型记忆管理框架）

**核心思想：** 模拟人类记忆层次，解决 LLM "上下文有限" 问题

#### 记忆层次

| 层次 | 说明 | 类似 OpenClaw 的 |
|------|------|-----------------|
| **VIP Memory** | 最重要的事实（角色、关键目标） | SOUL.md / MEMORY.md |
| **Core Memory** | 核心上下文（近期任务、当前状态） | MEMORY.md 精选条目 |
| **External Storage** | 归档的历史信息 | memory/YYYY-MM-DD.md + wiki/ |

#### 关键机制

1. **分层检索**：优先查 VIP → Core，miss 了再查 External
2. **动态上下文管理**：根据当前任务动态决定哪些记忆在窗口内
3. **遗忘触发**：信息按时间 + 使用频率双重维度淘汰

**电影小镇可以借鉴的：**
- 遗忘机制：当前 OpenClaw 没有自动遗忘，只有 promotion
- 动态上下文：根据任务类型动态调整记忆层级

---

### 2.3 对比总结

| 维度 | OpenClaw | GenericAgent | MemGPT |
|------|----------|-------------|--------|
| **记忆分层数** | 3层（日志/ curation/ wiki） | 5层（L0-L4） | 3层（VIP/Core/External） |
| **信息压缩策略** | 手动 promotion | 自动 crystallize | 使用频率衰减 |
| **遗忘机制** | ❌ 无 | ❌ 无（但有索引清理） | ✅ 使用频率 + 时间衰减 |
| **召回策略** | 语义搜索（memory_search） | 索引路由（Insight Index） | 分层检索优先 |
| **技能沉淀** | 手动写 SOP | 自动从执行路径提取 | 无 |
| **会话归档** | session-ingestion（原始索引） | Session Archive（提炼记录） | 无 |
| **Token 效率** | 中等 | 高（<30K） | 高 |

## 三、对电影小镇系统的改进建议

### 3.1 当前最大短板

**第一位：容量管理机制缺失**

- MEMORY.md 反复超限（100 行限制被多次提及）
- memory/*.md 无限增长，没有归档/淘汰策略
- 梦境索引（session-ingestion.json）持续膨胀，很多条目从未被真正使用

**第二位：Promoted 机制半自动化**

- 当前靠手动在 daily log 里写 `<!-- openclaw-memory-promotion:... -->` 标记
- 没有自动识别高价值条目的算法

**第三位：L3（梦境系统）和 L2（每日日志）的职责重叠**

- daily log 既是个股走势记录，也是工作日志，模糊了"情景记忆"和"语义知识"的边界

---

### 3.2 引入 GenericAgent L0-L4 的可行路径

#### 可快速实现的（1-2周）

| GenericAgent 层 | 对应 OpenClaw 现有机制 | 实现方案 |
|----------------|----------------------|---------|
| **L0 Meta Rules** | SOUL.md / AGENTS.md | 已有，无需改动 |
| **L1 Insight Index** | phase-signals.json / short-term-recall.json | 已有结构，重构索引逻辑 |
| **L2 Global Facts** | MEMORY.md | 扩展容量 + 增加自动压缩 |
| **L3 Task Skills/SOPs** | wiki/SOP/ + AGENTS.md rules | **重点建设**：从执行路径自动提取 SOP |
| **L4 Session Archive** | memory/.dreams/session-ingestion.json | **重点建设**：提炼归档记录而非存原始索引 |

#### 中期改进（1个月）

1. **建立遗忘机制**：
   - 对 memory/*.md 设置 TTL（如 30 天自动归档到 wiki/sources/）
   - 对 session-ingestion.json 设置容量上限（最多保留 N 条最近会话）

2. **建立自动 crystallize 流程**：
   - 每日复盘时自动提取"执行路径 → Skill 条目"写入 wiki/SOP/
   - 参照 GenericAgent 的"每次任务自动沉淀 Skill"理念

3. **分离情景记忆和语义知识**：
   - memory/*.md = 情景记忆（只保留 7-14 天）
   - wiki/ = 语义知识（永久积累）
   - MEMORY.md = 当前任务的上下文缓存

## 四、总结

### OpenClaw 记忆系统评价

**优点：**
- L0 角色层设计扎实，身份清晰
- Wiki 层（Karpathy 模式）很超前，外部知识和内部记忆分离
- 梦境系统的概念领先（短期召回 + 自动评分）

**最大不足：**
- 容量管理是致命短板——MEMORY.md 反复超限、日志无限增长
- 缺乏自动遗忘/promotion 机制，需要大量人工维护
- L3（梦境）和 L2（每日日志）的边界模糊

### 改进优先级

1. 🔴 **立即**：建立 MEMORY.md 自动压缩机制（每次 Session 启动时检查，超限自动promote 到 wiki/）
2. 🔴 **立即**：对 memory/*.md 设置 TTL（30 天后自动归档到 wiki/sources/）
3. 🟡 **短期**：实现从执行路径自动提取 SOP 的 crystallize 流程
4. 🟡 **短期**：重构 session-ingestion.json，区分"原始会话索引"和"提炼归档"
5. 🟢 **中期**：建立遗忘机制（使用频率 + 时间双重维度）
